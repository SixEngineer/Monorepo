OpenBridge 团队开发规范
1. 文档目的

为保证 OpenBridge 项目开发过程中的协作效率、代码质量和版本管理规范，团队统一采用本规范进行开发、提交、合并、测试和文档维护。

本规范适用于项目所有成员。

2. 项目协作原则

所有人都在统一 GitHub 仓库中协作开发。

不允许直接在主分支上随意修改代码。

每位成员在自己的功能分支上开发。

所有代码合并必须通过 PR（Pull Request）进行。

提交信息、分支命名、目录结构尽量统一。

每次开发都要保证“自己写的部分至少能运行或说明当前状态”。

发现问题及时记录到 Issue 或会议纪要中，不口头遗忘。

3. 仓库与分支管理规范
3.1 仓库说明

本项目采用单仓库（Monorepo）结构，统一管理后端、前端和文档资源。

推荐目录结构如下：

openbridge/
├── backend/      # Go 后端
├── frontend/     # Vue 前端
├── docs/         # 项目文档
├── scripts/      # 启动或辅助脚本
└── README.md
3.2 分支说明

项目统一使用以下分支结构：

main：稳定分支，用于保存最终可展示、可答辩、可发布版本

dev：开发主分支，用于集成日常开发成果

feature/*：功能分支，每位成员开发具体功能时使用

fix/*：问题修复分支

docs/*：文档修改分支

3.3 分支使用规则
main 分支

不允许直接提交代码

只允许通过 PR 从 dev 合并进入

用于保存阶段性稳定版本

dev 分支

作为团队主开发分支

所有功能开发完成后，先合并到 dev

日常联调、测试主要基于 dev

feature 分支

每位成员开发前从 dev 拉出自己的功能分支，例如：

feature/backend-init
feature/frontend-dashboard
feature/redirect-resolver
feature/quota-provider
fix 分支

用于修复具体问题，例如：

fix/login-bug
fix/api-response-error
docs 分支

用于文档、报告、说明文件修改，例如：

docs/readme-update
docs/meeting-notes
4. 开发流程规范
4.1 基本开发流程

每位成员开发任务时，统一按照以下流程进行：

先切换到 dev 分支并拉取最新代码

从 dev 创建自己的功能分支

在功能分支上完成代码编写

本地自测通过后提交代码

推送到远程仓库

发起 PR 合并到 dev

由至少 1 名成员进行 Review

Review 通过后合并

4.2 标准操作步骤
git checkout dev
git pull origin dev
git checkout -b feature/功能名

开发完成后：

git add .
git commit -m "feat: 添加某功能"
git push -u origin feature/功能名

然后在 GitHub 页面发起 PR，目标分支选择 dev。

4.3 合并规则

不允许跳过 PR 直接合并重要代码

PR 标题要清晰说明内容

合并前尽量确保没有明显冲突

合并后可删除已完成的功能分支

5. 提交规范
5.1 Commit 提交格式

统一采用：

类型: 简要说明

常用类型如下：

feat：新增功能

fix：修复问题

docs：文档修改

refactor：代码重构

style：格式调整，不影响逻辑

test：测试相关

chore：杂项修改、配置修改

5.2 提交示例
feat: 初始化后端服务
feat: 添加aria2 RPC适配
fix: 修复下载接口参数错误
docs: 更新项目说明文档
refactor: 重构quota provider接口
test: 增加下载链路测试
5.3 提交要求

一次提交尽量只做一件事

提交信息不要写成“修改一下”“更新”“test”

不提交明显不能运行的大量垃圾代码

临时代码、调试输出在合并前应清理

6. Pull Request 规范
6.1 PR 的作用

PR 用于请求将自己的代码合并到目标分支，同时供队友进行检查、讨论和 Review。

6.2 PR 标题格式

推荐格式：

[模块] 功能说明

例如：

[backend] 初始化HTTP服务
[frontend] 新增仪表盘页面
[quota] 添加容量统一模型
[resolver] 实现302跳转解析
6.3 PR 内容建议

PR 描述中至少写清楚：

本次修改做了什么

解决了什么问题

是否有未完成部分

是否需要队友重点关注某些代码

可参考模板：

## 本次修改内容
- 新增下载接口
- 增加基础参数校验
- 返回统一JSON格式

## 当前状态
- 本地已测试通过
- 仍未接入真实OpenList

## 需要关注
- 接口返回字段是否需要调整
6.4 Review 规范

进行代码 Review 时，重点关注：

功能逻辑是否正确

命名是否清晰

是否会影响他人模块

是否存在明显冗余或错误

文档是否需要同步更新

7. 代码规范
7.1 通用要求

命名尽量清晰，不使用无意义变量名

代码结构尽量简单，避免一层套一层

重复逻辑尽量提取复用

重要逻辑加注释

配置与业务代码尽量分离

敏感信息不写死在代码里

7.2 后端规范（Go）

接口返回统一 JSON 格式

错误处理尽量明确

配置文件独立管理

模块职责清晰，避免所有代码都写在 main.go

日志信息尽量保留必要上下文

建议后端逐步拆分为：

backend/
├── cmd/
├── internal/
│   ├── adapter/
│   ├── resolver/
│   ├── quota/
│   ├── provider/
│   └── config/
7.3 前端规范（Vue）

页面、组件、状态管理分离

公共组件尽量复用

请求接口统一封装

错误提示明确，不要静默失败

页面风格尽量统一

8. 文档规范
8.1 必须维护的文档

项目至少维护以下文档：

README.md：项目简介、运行方式、技术栈

docs/meeting-notes/：会议记录

docs/api.md：接口说明

docs/architecture.md：系统架构说明

docs/task-plan.md：任务分配与进度安排

8.2 会议记录要求

每次会议建议记录以下内容：

会议时间

参与成员

当前进展

存在问题

本周任务

下次会议目标

8.3 文档更新原则

代码改动较大时，应同步更新相关文档，避免“代码变了，文档没变”。

9. Issue 与任务管理规范
9.1 Issue 用途

Issue 用于记录：

待开发功能

Bug

优化建议

待讨论问题

9.2 Issue 标题建议
[Backend] 初始化服务框架
[Frontend] 完成登录页
[Bug] 下载接口返回为空
[Docs] 补充系统架构图说明
9.3 任务分配建议

每个 Issue 尽量指定：

负责人

优先级

截止时间

关联分支或 PR

10. 环境与配置规范
10.1 环境统一原则

团队成员应尽量统一开发环境版本，避免“我这里能跑，你那里不能跑”。

建议统一：

Go 版本

Node.js 版本

npm 或 pnpm 版本

Git 版本

10.2 配置文件规则

本地开发配置不要直接提交真实账号、密码、token

使用示例配置文件，例如：

.env.example
config.example.yaml

真实敏感配置写入本地环境，不上传仓库

11. 测试与联调规范
11.1 基本要求

每位成员提交代码前应尽量完成最基本的自测。

自测包括但不限于：

能否正常运行

接口是否可访问

页面是否能打开

是否影响已有功能

11.2 联调要求

涉及多个模块联动时，应提前沟通接口格式，避免联调时重复修改。

例如：

前端与后端先约定接口字段

下载模块与 quota 模块先约定返回结构

provider 模块先定义统一抽象接口

11.3 Bug 处理原则

发现 Bug 后：

先记录现象

尽量说明复现步骤

标记影响范围

指定修复负责人

修复后记录结果

12. 安全与保密规范

虽然本项目是课程项目，但仍需注意以下问题：

不在仓库中上传真实密码、token、cookie

不公开敏感调试信息

不在 README 中泄露本地服务地址和密钥

不将抓包得到的敏感信息随意共享

13. 团队沟通规范
13.1 日常沟通

重要决定应形成文字记录

模块接口改动应提前通知相关成员

遇到卡住的问题不要长期闷头拖延，应及时反馈

13.2 会议建议

建议每周至少进行一次小组会议，内容包括：

上周完成情况

当前存在问题

本周任务分工

需要协调的接口或联调事项

14. 课程项目特别说明

本项目属于课程工程实践项目，因此规范目标是：

保证项目能顺利推进

保证答辩时材料完整

保证每位成员都有清晰分工和实际贡献

保证代码、文档、过程都能体现软件工程方法

因此，规范执行以“清晰、统一、可落地”为原则，不追求企业级复杂流程，但应尽量避免无序开发。

15. 附：团队约定简版

为方便执行，团队统一遵守以下简版约定：

不直接改 main

日常开发基于 dev

每人使用自己的 feature/* 分支

合并代码必须发 PR

Commit 信息要规范

写完功能至少先自测

改接口要通知相关成员

重要决定写入文档或会议记录

敏感信息不上传仓库

遇到问题及时沟通，不拖延
